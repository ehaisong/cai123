
# H5 彩票分析预测内容付费平台

## 一、产品概述

参考截图样式（"小吴说菜"店铺），搭建一个移动端 H5 内容付费平台。三种角色，覆盖完整业务闭环：

```text
┌─────────────────────────────────────────────────┐
│          平台管理员 (Admin Console)              │
│  审核商家入驻 / 内容审查 / 财务结算 / 提现审批    │
└─────────────────────────────────────────────────┘
                       ↑
┌─────────────────────────────────────────────────┐
│          商家后台 (Merchant H5)                  │
│  发布预测商品 / 设价 / 收益统计 / 二维码 / 提现   │
└─────────────────────────────────────────────────┘
                       ↑
┌─────────────────────────────────────────────────┐
│          买家 / 代理 (Buyer H5)                  │
│  浏览/购买预测 / 申请代理 / 推广分成 / 钱包       │
└─────────────────────────────────────────────────┘
```

## 二、用户角色与核心流程

### 1. 买家（普通用户）
- 注册/登录（手机号 + 验证码）
- 进入商家店铺首页（参考图1）：平台公告、栏目筛选（全部/福彩3D/六合彩/足彩）、商品列表（标题、★强烈推荐★、价格、发布时间）
- 点击商品进入详情页（参考图3/4）：发布时间、公开时间、免责声明、付费内容（解锁后显示）、往期记录（显示中奖/未中印章）
- 钱包充值（余额制，参考图5）→ 用余额购买商品
- "我的"页：余额 / 资金明细 / 订单 / 反馈 / 联系客服
- 申请成为代理 → 获得专属推广二维码

### 2. 商家
- 提交入驻申请（参考图6）：姓名、手机号+验证码、微信号、粉丝数、公众号、申请说明
- 审核通过后进入商家后台
- 发布商品：彩种类型、期号、标题、推荐标签、价格、发布时间、公开时间、付费内容、往期记录
- 商品管理：上下架 / 编辑 / 标记中奖结果
- 数据看板：销售额、订单数、代理数、本月收益
- 提现申请、专属推广二维码、店铺设置

### 3. 平台管理员
- 商家入驻审核（通过/驳回 + 理由）
- 商家与商品管理（可下架违规商品）
- 用户管理 / 代理关系树查看
- 财务结算：手动给买家钱包充值、平台抽成配置、商家提现审批
- 公告管理（首页"平台公告"位）

## 三、多级分销机制（两级）

```text
买家A → 通过商家M的二维码进入并购买
        → 平台记录："A 由 M 直接获客"

买家A 申请成为代理后获得自己的推广二维码（绑定到 M）
买家B 扫A的码注册并购买商品（任意商家M的商品）
        → A 获得一级分成（如 10%）
        → 若 B 也是代理，B 推荐的 C 购买
          → B 获一级分成、A 获二级分成（如 5%）

分成比例由平台管理员全局配置，单次购买分账写入 commission_records
```

> 控制在两级，避免触及国内传销红线。

## 五、数据模型（核心表）

- `profiles`：用户基本信息（昵称、头像、手机号、用户编号）
- `user_roles`：角色（buyer / agent / merchant / admin），独立表防越权
- `merchants`：商家档案（姓名、微信、粉丝数、申请说明、状态、店铺信息）
- `merchant_applications`：入驻申请历史
- `lottery_categories`：彩种（福彩3D / 六合彩 / 足彩）
- `products`：商品（商家、彩种、期号、标题、推荐标签、价格、发布时间、公开时间、状态、中奖结果）
- `product_contents`：付费内容（购买后才可读取，RLS 严控）
- `product_history`：往期记录（期号、内容、中奖与否）
- `orders`：订单（买家、商品、商家、价格、推荐人agent_id、状态）
- `wallets`：用户余额
- `wallet_transactions`：充值/消费/分成/提现明细
- `agent_relations`：代理关系（agent_id → upline_id → merchant_id）
- `commission_records`：每笔订单的分成快照（一级/二级/平台）
- `commission_config`：分成比例配置
- `withdrawals`：提现申请
- `announcements`：平台公告
- `feedback`：反馈

所有表启用 RLS，敏感操作走 server function 验证身份。

## 六、技术要点

- **框架**：TanStack Start + React 19（项目已就绪）
- **后端**：Supabase（已连接）。鉴权、RLS、数据库、Edge Functions、Storage（商家头像/店铺图）
- **样式**：Tailwind v4 + shadcn/ui，移动端 H5 优先（最大宽度 480px 居中容器）
- **设计风格**：参考截图——浅灰背景（#f5f5f7）、白色卡片、橙红主色（公告/价格/推荐标签）、绿色 CTA（充值/支付按钮）、蓝色（管理操作）
- **支付**：第一版采用"钱包余额"模型 → 充值入口预留"微信/支付宝"按钮（先跳转占位页，后续你自行接入支付服务商 SDK，对应回调写入 `wallet_transactions`）。订单结算全部走余额扣款，逻辑闭环完整。
- **二维码**：使用 `qrcode` 库本地生成，链接形如 `/shop/$merchantId?ref=$agentCode`，注册时自动绑定上级
- **手机验证码**：留 server function 接口（dev 模式打印控制台 / 任意 6 位通过），后续替换为短信服务商
- **图片资源**：印章（中奖/未中）、占位头像等用 SVG 内联或放 `src/assets`

## 七、分阶段实施

由于涉及三端 + 多级分销 + 钱包结算，工作量较大，建议分两次构建：

**第一批（本次实施 — MVP 主干）**

6. 二维码生成 + 推荐人绑定 + 单级分成入账

**第二批（后续迭代）**
7. 二级分成、代理下级树
8. 商家提现流程完整化、资金明细筛选
